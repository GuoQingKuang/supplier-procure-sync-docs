## 五、开发规范与注意事项

### 4.1 分层架构

```
Controller → Service → Dao(extends ServiceImpl<Mapper, Entity>) → Mapper(extends BaseMapper<Entity>)
```

- Dao 继承 `ServiceImpl<Mapper, Entity>`，使用 `lambdaQuery()` 构建查询
- Controller 用 `@Valid` 触发参数校验
- 异常用 `ThrowUtils.throwIf(condition, ErrorCode, message)`

### 4.2 认证机制

- **供应商登录**：`POST /auth/login`，token 中 `userId` claim 存 `supplierId` 字符串，`role` claim 存 `"supplier"`
- **管理员登录**：企业微信 OAuth2，走 `GET /oauth2/login`（`OAuth2Controller`），由 Spring Security OAuth2 回调处理
  - 回调中通过 `given_name`（wxUserId）查 `fscut_user_ref` 表，生成 token 后重定向到前端
  - token 中携带 userId（hrmUserId）
- **JWT 认证**：`JwtInterceptor` 拦截请求，从 `Authorization` Header 或 `auth_code` 参数提取 token
  - `role=supplier`：只把 supplierId 存入 ThreadLocal，跳过企业微信查询
  - `role=admin`：查企业微信用户信息，存入 userId 和 name
- **拦截器放行路径**：`/auth/**`、`/admin/**`（临时放行 TODO）、`/wx/**`、`/static/**`、`/webjars/**`、`/oauth2/login`

### 4.3 DuplicateKeyException 处理

`GlobalExceptionHandler` 统一捕获 `DuplicateKeyException`，返回 `40000 + "用户名已存在"`，Service 层无需 try-catch。
创建供应商账号时依赖数据库 `username` 唯一索引保证并发安全。

### 4.4 JSON 字段处理

`delivery_notice` 表的以下字段用 `String` 类型直接存 JSON 字符串，不用 TypeHandler：
- `materialData`：原始物料明细 JSON 数组
- `kingdeePushParams`：下推 API 传参留样
- `kingdeeUpdateParams`：修改 API 传参留样

读写时用 `JSON.parseObject()` / `JSON.parseArray()` 手动处理。

### 4.5 金蝶对接要点

- **数据查询**：优先查金蝶 SQL Server（`@DS("erp")`），API 调用作为备选
- **写入操作**：必须走金蝶 API（下推、保存、作废）
- **SDK 线程安全**：`K3CloudApi` 非线程安全，每次调用应新建实例
- **收料单标识**：Push 成功后 `SuccessEntitys[0].Id` 是收料通知单内码，存入 `delivery_notice.kingdee_receive_bill_id`
- **EntryIds 规则**：采购订单分录内码全局唯一，多个用逗号分隔，金蝶按供应商+收货方自动分组
- **批量下推**：提交发货时所有分录合并为一次 Push 调用，生成一张收料通知单
- **SDK 封装**：业务代码通过 `KingdeeApiService` 调用（Push/Save/Cancel），不直接使用 SDK
- **Save API 字段命名**：`NeedUpDateFields` 中不带 F 前缀，Model 中带 F 前缀（如 `FActReceiveQty`）

### 4.6 状态枚举

**delivery_notice.status**：
| 值 | 含义 |
|----|------|
| 1 | 发货中 |
| 2 | 已发货 |
| 3 | 已作废 |
| 4 | 异常 |

**金蝶采购订单状态**：
| 字段 | 有效值 | 含义 |
|------|--------|------|
| FDocumentStatus | C | 已审核 |
| FCancelStatus | A | 未作废 |
| FCloseStatus | A | 未关闭 |
| FMRPFreezeStatus | A | 正常 |
| FMRPCloseStatus | A | 正常 |
| FMRPTerminateStatus | A | 正常 |

### 4.7 密码存储

供应商账号密码采用明文存储（走 HTTPS），后端登录时直接字符串比对。

### 4.8 送料通知单号生成

格式：`SN + yyyyMMdd + 4位序号`，如 `SN202403200001`
建议用 Redis INCR 保证分布式环境下唯一性。

### 4.9 异常处理

发货流程涉及多次金蝶 API 调用，需要 `@Transactional(rollbackFor = Exception.class)` 管理本地事务。
失败时 catch 中调 Cancel 作废金蝶收料单 + 更新本地状态 + 记录 fail_reason。

### 4.10 跨系统数据追溯链路

```
采购订单分录内码(FEntryID)
    → Push API EntryIds 参数
    → 生成收料通知单(FID) + 收料单分录内码数组(EntryIds.FDetailEntity)
    → 收料单分录.FSRCENTRYID = 原始采购订单分录内码（可追溯）
    → 本地 delivery_notice 表 kingdee_receive_bill_id = 收料单FID
```

### 4.11 降级策略

送料通知单列表查询时，若金蝶 SQL Server 不可用：
- 成功记录返回空数组
- 失败记录直接从 `material_data` JSON 读取
- 附加 warning 提示前端
