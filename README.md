# 供应商物料协同平台 - 架构与设计文档

面向供应商的采购订单协同平台，对接金蝶 ERP 系统，替代人工在金蝶系统操作的低效流程，已上线运行，预估节省半个人力成本（约 10 万元/年）。

## 技术栈

| 分类 | 技术 |
|------|------|
| 后端框架 | Spring Boot 3.1.7、Java 17 |
| ORM | MyBatis-Plus 3.5.5 |
| 数据源 | dynamic-datasource 4.2.0（MySQL + SQL Server 多数据源） |
| 缓存 | Redis + Redisson 3.10.6 |
| 认证 | OAuth2 授权码登录 + JWT |
| 文件存储 | MinIO 8.6.0 |
| ERP 对接 | 金蝶 WebAPI SDK 8.2.0 |
| 监控 | Apollo 配置中心 + Prometheus |

## 核心设计

### 预发货单设计
将"填写数据"与"推送金蝶"解耦，供应商可在预发货阶段随意修改，确认后才生成正式单推金蝶，降低误操作成本。

### 跨系统容错机制
发货流程涉及本地落库 → 金蝶 Push → 金蝶 Save 三步跨系统操作。Push 成功但 Save 失败时，自动调用 Delete 删除已创建的收料单，标记本地状态为已作废，保证跨系统数据最终一致性。

### 多数据源事务隔离
MySQL 业务库与 SQL Server ERP 库通过 dynamic-datasource 动态切换。ERP 查询使用 `REQUIRES_NEW` 独立事务，通过 `self` 代理调用解决同类内部调用不走 AOP 代理导致数据源不切换的问题。

### 分录映射处理
金蝶下推后，采购订单分录与收料单分录的对应关系不按顺序，需通过 `T_PUR_RECEIVEENTRY_LK` 关联表查询映射，确保交货数量设置到正确的分录上。

## 关键接口

| 接口 | 说明 |
|------|------|
| `POST /delivery-notice/submit` | 提交发货 |
| `GET /delivery-notice/list` | 送料通知单列表 |
| `GET /delivery-notice/detail` | 送料通知单详情 |
| `POST /delivery-notice/submit-header` | 提交表头 + 审核 |
| `GET /order/list` | 采购订单列表 |
| `POST /order/export` | 采购订单导出 |

## 业务流程

```
供应商登录 → 查看采购订单 → 创建预发货单(草稿/可修改) → 确认转正
    → 本地落库(发货中) → 金蝶 Push 下推收料单 → 金蝶 Save 修改交货数量
    → 全部成功(已发货) / 失败(补偿删除收料单 + 已作废)
    → 填写表头字段 → 金蝶 Submit 提交 → 审核中
```
