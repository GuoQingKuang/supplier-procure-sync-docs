## 一、供应商认证模块

### 1.1 供应商登录

**接口**: `POST /auth/login`

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 正常 | 输入正确的用户名和密码 | 返回 `code=0`，响应中含 `token` 和 `supplierInfo`，跳转到首页 |
| 错误密码 | 输入正确用户名、错误密码 | 返回 `code≠0`，提示"用户名或密码错误" |
| 错误用户名 | 输入不存在的用户名 | 返回 `code≠0`，提示"用户名或密码错误"（与上条提示一致） |
| 空用户名 | 用户名为空，直接点击登录 | 前端应拦截（必填校验） |
| 空密码 | 密码为空，直接点击登录 | 前端应拦截（必填校验） |
| 用户名含空格 | 用户名输入 `" abc "`（前后空格） | 后端返回参数错误，提示"用户名不能包含空格" |
| 密码含空格 | 密码输入 `"123 456"`（中间空格） | 后端返回参数错误，提示"密码不能包含任何空白字符" |
| 用户名过短 | 用户名输入 1 位字符 | 后端返回参数错误，提示"用户名长度必须在2~50之间" |
| 密码过短 | 密码输入 5 位字符 | 后端返回参数错误，提示"密码长度必须在6~64之间" |
| 用户名超长 | 用户名输入 51 位字符 | 后端返回参数错误 |
| 密码超长 | 密码输入 65 位字符 | 后端返回参数错误 |
| 禁用账号登录 | 用已禁用的供应商账号登录 | 返回 `code≠0`，提示"账号已被禁用" |
| 特殊字符用户名 | 用户名输入 `<script>alert(1)</script>` | 后端返回参数错误（不匹配正则） |
| 换行符注入 | 用户名输入 `"abc\r\n"` 或密码含换行 | 后端返回参数错误 |
| 单设备登录 | 设备 A 登录 → 设备 B 用同一账号登录 → 设备 A 发起请求 | 设备 A 的请求返回 token 失效/被踢提示，跳转登录页 |

**验证要点**:
- 登录成功后，后续请求的 `Authorization` Header 是否正确携带 `Bearer <token>`
- 单设备登录：两个设备同时登录同一账号，先登录的设备下次请求应被踢下线

---

### 1.2 退出登录

**接口**: `POST /auth/logout`

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 正常 | 点击退出登录 | 返回 `code=0`，跳转登录页 |
| 退出后再次请求 | 退出后用旧 token 访问接口 | 返回 token 失效，跳转登录页 |
| 退出后重新登录 | 用同一账号重新登录 | 登录成功，获取新 token |
| 无 token 退出 | 不携带 token 调用退出接口 | 返回 token 无效错误 |

## 二、采购订单列表

**接口**: `GET /order/list`（需 supplier token）

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 正常 | 进入订单列表页，不输入筛选条件 | 返回分页数据，显示当前供应商的采购订单 |
| 分页 | 翻到第 2 页 | 显示下一页数据，`total` 正确 |
| 按单据编码筛选 | 输入部分单据编码搜索 | 只返回匹配的订单 |
| 按物料编码筛选 | 输入部分物料编码搜索 | 只返回匹配的订单 |
| 按日期范围筛选 | 选择起始日期和截止日期 | 只返回范围内的订单 |
| 日期起始 > 截止 | 起始日期晚于截止日期 | 返回空数据（SQL 逻辑） |
| 无数据的供应商 | 用无采购订单的供应商账号查看 | 返回空列表，`total=0` |
| 超长搜索词 | 输入 200 位字符的搜索词 | 正常返回空或匹配数据，不报错 |
| 特殊字符搜索 | 搜索词输入 `' OR 1=1 --` | 正常返回，不触发 SQL 注入 |
| 分页参数异常 | 手动传 `current=-1` 或 `pageSize=999999` | 正常处理或返回错误，不崩溃 |
| token 过期 | token 过期后刷新页面 | 跳转登录页 |
| 无 token 访问 | 不携带 token 直接调接口 | 返回未登录错误 |

## 三、提交发货

**接口**: `POST /delivery-notice/submit`（需 supplier token）

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 正常提交 | 勾选订单，输入合法的发货数量（≤ remainQty），提交 | 返回 `code=0`，响应中含 `noticeNo`，列表出现该通知单 |
| 发货数量 = 剩余数量 | 发货数量恰好等于剩余数量 | 提交成功 |
| 发货数量 > 剩余数量 | 发货数量超过剩余数量 | 返回参数错误，提示"发货数量超过剩余收料数量" |
| 发货数量 = 0 | 发货数量输入 0 | 返回参数错误（@NotNull 校验）或业务错误 |
| 发货数量 = 负数 | 发货数量输入 -1 | 返回参数错误 |
| 物料列表为空 | 不勾选任何物料直接提交 | 返回"发货物料列表不能为空" |
| 无效 entryId | 传一个不存在的 entryId | 返回"分录内码不存在或已无剩余收料数量" |
| 已下推的订单 | 对已下推过的订单再次提交 | 金蝶返回失败，本地通知单标记为"已作废" |
| 多物料同时提交 | 勾选 3 条不同订单分录，同时提交 | 全部成功或全部失败（原子性） |
| 重复提交（快速双击） | 提交按钮快速双击 | 第一次成功，第二次应报错或重复提交被拦截 |
| 超长字段测试 | 物料名称/编码传 200 位字符 | 正常处理或返回校验错误 |
| 特殊字符 | 物料名含 `' " < > &` 等特殊字符 | 正常处理，不触发 XSS/SQL 注入 |

**验证要点**:
- 提交成功后，送料通知单列表中立即出现该记录
- 提交失败后，列表中不应出现成功记录（或显示为"已作废"）
- 查看 Network 面板，确认提交接口请求体中 `materialSpec` 字段名正确（不是 `spec`）

## 四、送料通知单列表

**接口**: `GET /delivery-notice/list`（需 supplier token）

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 正常 | 进入通知单列表页 | 分页显示成功记录，失败记录全量显示 |
| 分页 | 翻到第 2 页 | 成功记录分页正确，失败记录始终全量 |
| 按通知单号筛选 | 输入部分通知单号（如 `SN2026`） | 只匹配的通知单出现 |
| 按物料编码筛选 | 输入物料编码搜索 | 含该物料的通知单出现（成功记录和失败记录都应过滤） |
| 按发货时间范围筛选 | 选择起始和截止时间的范围 | 只返回范围内的通知单 |
| 时间起始 = 截止 | 起始和截止选同一天 | 返回当天的记录 |
| 无数据 | 新供应商首次查看 | 返回空列表和空失败记录 |
| 金蝶不可用时 | 模拟金蝶系统不可用 | 成功记录列表为空，`warning` 字段有提示信息 |
| 超长通知单号搜索 | 输入 100 位字符的通知单号 | 正常返回空，不报错 |
| 特殊字符搜索 | 搜索词含 SQL 注入字符 | 正常返回，不触发注入 |

**验证要点**:
- `notices` 字段含分页的成功记录
- `failedNotices` 字段含全量失败记录（不分页）
- `warning` 字段在金蝶不可用时有提示值
- 成功记录的 `items` 含金蝶查询的物料详情
- 失败记录的 `items` 含 `material_data` JSON 解析的物料明细
- `status` 状态码对应正确：1=发货中，2=已发货，3=已作废，4=异常

## 十、管理端 - 更新供应商账号状态

**接口**: `PUT /admin/supplier-account/{id}/status`（需 admin token）

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 禁用账号 | 传 `{"status": 0}` | 状态变为 0，账号被禁用 |
| 启用账号 | 传 `{"status": 1}` | 状态变为 1，账号可登录 |
| 禁用已在线账号 | 禁用一个正在登录的供应商账号 → 该供应商发起请求 | **供应商的请求应被踢下线**（token 加入黑名单） |
| 传非法 status | 传 `{"status": 99}` | 返回"状态值只能为0或1" |
| 不传 status | 传 `{}` | 返回"状态不能为空" |
| 不存在的 ID | 传一个不存在的 id | 返回"账号不存在" |
| 无 admin token | 用 supplier token 调用 | **应返回 403 禁止访问** |
| 自己禁用自己 | 用 supplier token 调用此接口 | **应返回 403**（角色不对） |

**验证要点**（禁用已在线账号场景）:
1. 供应商 A 登录成功
2. 管理端将供应商 A 的账号状态设为禁用
3. 供应商 A 刷新页面或发起任何请求
4. 供应商 A 应收到 token 被踢/失效的提示，跳转登录页
5. 供应商 A 尝试重新登录，应返回"账号已被禁用"

## 十三、异常场景测试

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 无效 token | Header 传伪造的 JWT token | 返回"Token无效" |
| token 过期 | 等 token 自然过期后发起请求 | 返回 token 失效，跳转登录页 |
| token 被篡改 | 修改 JWT token 中的某几个字符 | 验签失败，返回"Token无效" |
| 无 Header 访问 | 不携带 `Authorization` 访问需认证接口 | 返回未登录错误 |
| 网络断开 | 操作过程中断开网络 | 前端显示网络错误 |
| 金蝶服务不可用 | 金蝶系统宕机时提交发货 | 返回失败，通知单标记为"已作废" |
| 并发提交 | 两个设备同一供应商同时提交发货 | 一个成功，一个失败（金蝶端控制） |

## 测试通过标准

1. 所有"正常"场景的测试用例全部通过
2. 所有"越权"场景均返回 403
3. 所有"异常"场景均有合理的错误提示，无 500 崩溃
