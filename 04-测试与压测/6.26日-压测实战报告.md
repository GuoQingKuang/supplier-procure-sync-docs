# ProcureSync 采购协同系统 — 上线前压测实战报告

## 1. 压测背景

### 1.1 系统架构简介

**技术栈：** Spring Boot 3.1.7 + Java 17 + MyBatis-Plus 3.5.5

**调用链路：**

```
前端 → Controller → Service → MySQL（主库业务数据）
                        → SQL Server（金蝶 ERP 数据）
                        → Redis（Token 管理 / 防爆破）
                        → 金蝶 K3 Cloud API（Push / Save / Submit）
```

**核心接口：**
- 提交发货：调金蝶两个 API（Push + Save）
- 送料通知单列表：双库查询
- 采购订单列表：8+ 表 JOIN 查询

### 1.2 为什么压测

系统预期上线后并发 300，涉及外部金蝶 ERP API 调用和多数据源查询，上线前需要验证：
- 系统能否承受预期并发压力
- 外部依赖（金蝶 API）对性能的影响
- 并发场景下的数据一致性

### 1.3 压测目标与验收标准

| 指标 | 目标值 |
|------|--------|
| 并发用户数 | 200（测试服务器预期 300，本机先压 200） |
| P95 响应时间 | submit < 2s，submit-header < 5s |
| 错误率 | < 1% |
| 超发/并发安全 | 0 发生 |

### 1.4 测试环境

| 组件 | 配置 |
|------|------|
| 压测工具 | JMeter（本机） |
| 被测应用 | 本机 Docker 部署 |
| 中间件 | MySQL / Redis / SQL Server 均在测试服务器 |
| 应用端口 | 8657 |

## 2. 压测方案设计

### 2.1 工具选型

**工具：JMeter**

**选型依据：**
- 成熟的开源压测工具，支持线程组阶梯加压
- CSV Data Set Config 方便构造不同测试数据
- 内置 Aggregate Report 提供 P50/P95/错误率等关键指标
- 支持自定义 HTTP Header（Bearer Token 认证）

### 2.2 接口优先级分级

| 优先级 | 接口 | 依据 |
|--------|------|------|
| P0 | 提交发货（/delivery-notice/submit） | 核心写操作，涉及金蝶 Push API 调用 |
| P0 | 提交审核（/delivery-notice/submit-header） | 核心写操作，涉及金蝶 Save + Submit 两次 API 调用 |
| P1 | 采购订单列表 | 高频读操作，8+ 表 JOIN |
| P2 | 送料通知单列表 | 双库查询 |

### 2.3 测试数据构造策略

- 使用 CSV Data Set Config 读取 200 条测试数据，每个线程取一条不同数据
- 提交发货场景：200 个不同采购订单各一个物料行，模拟 200 家供应商同时发货
- 提交审核场景：同样 200 个不同通知单并发提交
- **JSON 转义踩坑：** CSV 中物料名称含反斜杠 `\` 和双引号 `"`，导致 JMeter 拼接的 JSON 解析失败。在生成脚本中加了 `.replace('"', "'").replace("\\", "\\\\")` 修复

### 2.4 压测场景设计

- **正常流量场景：** Thread Group 200 线程，Ramp-Up 10s 逐步启动
- **竞争条件场景：** 早期用同一采购订单的不同物料行做并发，触发金蝶网控冲突（"同一采购订单的同一单据正在被XXX用户修改"），后改为 200 个不同采购订单解决
- **高并发冲击场景：** Ramp-Up 从 10s → 5s → 1s 逐步加压，观察响应时间陡增情况

### 2.5 环境准备与配置

- JMeter 本机安装，通过 HTTP 请求压测本机 8657 端口
- 被测应用 Docker 部署在测试服务器
- HTTP Header Manager 带 `Authorization: Bearer <token>`
- 监听器：View Results Tree（查错误）、Aggregate Report（看指标）

### 3.2 逐个接口压测详情

#### 3.2.1 提交发货接口（P0 - /delivery-notice/submit）

**测试场景：** 200 个不同采购订单各一个物料行，模拟 200 家供应商同时发货，调金蝶 Push API 下推收料通知单

**阶梯数据：**

| 线程数 | Ramp-Up | 平均响应时间 | P50 | P95 | P99 | QPS | 错误率 |
|--------|---------|-------------|-----|-----|-----|-----|--------|
| 200 | 10s | 988 ms | 942 ms | 1439 ms | 1642 ms | ~28 | 0% |
| 200 | 5s | 4190 ms | - | - | - | ~4.8 | 0% |
| 146 | 1s | 10846 ms | - | - | - | ~1.8 | 0% |

**现象分析：**
- Ramp-Up 10s 时平均响应时间 < 1s，P95 1.4s，QPS 28，表现正常
- Ramp-Up 5s 时响应时间陡增至 4.2s，QPS 下降到 4.8
- Ramp-Up 1s 时响应时间突破 10s，QPS 仅 1.8，大量线程排队
- **结论：** 瓶颈在金蝶 API 调用（同步阻塞等待外部系统响应），不是本地数据库或业务逻辑

#### 3.2.2 提交审核接口（P0 - /delivery-notice/submit-header）

**测试场景：** 200 个通知单并发提交，后端调金蝶 Save + Submit 两次 API

**阶梯数据：**

| 线程数 | Ramp-Up | 平均响应时间 | P50 | P95 | P99 | Min | QPS | 错误率 |
|--------|---------|-------------|-----|-----|-----|-----|-----|--------|
| 200 | 1s | 2768 ms | 3155 ms | 3308 ms | 3339 ms | 18 ms | 7.5 | 0% |

**现象分析：**
- 平均响应时间 2.8s，比 submit 的 10.8s 快，因为只调 Save + Submit 两次 API，不调 Push
- Min 18ms 是异常值，因网控冲突导致金蝶快速返回错误，没走完整流程。实际场景不会出现

### 3.3 踩坑记录

#### 踩坑 1：金蝶网控冲突导致并发失败

- **问题现象：** 早期测试用同一采购订单的不同物料行做并发，金蝶报错"同一采购订单的同一单据正在被XXX用户修改，请稍后再试"
- **排查过程：** 查看 View Results Tree 发现金蝶返回的错误信息，确认是金蝶系统内部的网控机制（防止同一单据被多人同时修改）
- **根因分析：** 金蝶对同一采购订单做了并发锁，多个请求操作同一订单会触发网控拦截
- **解决方案：** 改成 200 个不同采购订单各一个物料行，模拟 200 家供应商同时发货。实际生产也不会出现这个问题，因为供应商只能操作自己的订单
- **验证结果：** 修改后错误率降为 0%

#### 踩坑 2：CSV 中特殊字符导致 JSON 解析失败

- **问题现象：** JMeter 发送请求时报 JSON 解析错误
- **排查过程：** 查看请求体发现物料名称字段含反斜杠 `\` 和双引号 `"`，破坏了 JSON 结构
- **根因分析：** CSV 数据中的特殊字符未转义，直接拼接到 JSON 字符串中导致格式错误
- **解决方案：** 在生成 CSV 的脚本中加了 `.replace('"', "'").replace("\\", "\\\\")` 转义处理
- **验证结果：** JSON 解析正常，请求成功发送

#### 踩坑 3：submit-header 权限校验拦截

- **问题现象：** 压测 submit-header 接口时大量请求被拦截返回权限错误
- **排查过程：** 查看日志发现压测用的 token 对应的 supplier_id 和测试数据中的 supplier_id 不匹配
- **根因分析：** 这是正常的业务权限校验逻辑，但压测数据构造时没注意 token 和数据的一致性
- **解决方案：** 临时注释了权限校验跑完压测，之后已恢复
- **验证结果：** 压测顺利完成，代码已恢复权限校验

## 4. 发现的问题与优化

### 4.1 性能瓶颈

#### 4.1.1 数据库连接池瓶颈

- **配置：** Druid 连接池，`max-active` 默认 8
- **现象：** 200 并发时连接池被打满，其他线程排队等待数据库连接
- **优化方案：** 计划在 Apollo 配置中将 `max-active` 调整到 50，`initial-size` 设为 10

#### 4.1.2 应用线程池瓶颈

- **配置：** 核心线程数 = 13，最大线程数 = 25，队列容量 50，拒绝策略 CallerRunsPolicy
- **硬件限制：** 本地 CPU i5-11400（6核12线程）
- **现象：** 200 并发时最多 25 个线程并行执行，超出队列容量的请求由主线程执行（CallerRunsPolicy），进一步拖慢响应
- **分析：** Ramp-Up 1s 时响应时间突破 10s，除了金蝶 API 慢的影响，线程池排队也是重要因素

#### 4.1.3 金蝶 API 超时影响

- 瓶颈主要在金蝶 API 调用（同步阻塞等待外部系统响应），不是本地数据库或业务逻辑
- Ramp-Up 从 10s 降到 1s 时，平均响应时间从 988ms 陡增至 10846ms，翻了 10 倍
- **金蝶 API 单次调用耗时估算：** 从整体响应时间推断，submit（调 1 次金蝶 API）平均 1s，submit-header（调 2 次）平均 2.8s，金蝶单次 API 调用约 1~1.5 秒
- **后续可改进：** 用 Arthas trace 命令精确统计 `KingdeeApiService.push()` / `updateReceiveBillHeader()` / `submit()` 各方法的耗时

### 4.2 并发安全隐患

#### 4.2.1 超发风险评估

- **本次压测未做专门的超发场景测试**
- 金蝶侧有网控冲突保护，多个请求同时操作同一采购订单时，金蝶会自动拦截并返回冲突提示
- 实际生产环境中供应商只能操作自己的通知单，不会出现同一订单的并发操作
- **结论：** 超发风险在业务层面已经被隔离

## 5. 经验总结

### 5.1 如果重新做会改进什么

- **提前加连接池和线程池监控：** 压测时应该同时观察连接池活跃连接数、线程池队列长度，而不是只看 JMeter 结果
- **拆分金蝶 API 耗时：** 用 Arthas trace 命令统计每个金蝶 API 调用的耗时，精确找到慢的环节
- **阶梯式加压更科学：** 应该从 50 并发开始逐步递增，画出一条并发-响应时间曲线，而不是直接跳到 200
- **提前配置好监控：** 本地压测可以简单，但正式压测服务器应该配合 Grafana + Prometheus 或 Arthas 做全链路观察

### 5.2 对系统架构的反思

- **外部 API 是最大瓶颈：** 系统响应时间主要取决于金蝶 API 的调用次数和耗时，而不是本地业务逻辑。submit 接口调一次金蝶平均 1 秒，submit-header 调两次就变成 3 秒
- **连接池默认配置撑不住高并发：** Druid 默认最大连接数只有 8，线程池最大 25，面对 200 并发时连接池和线程池都会成为排队瓶颈
- **同步调用的局限性：** 当前金蝶 API 是同步阻塞调用，并发越高排队越严重。如果引入异步化（如先入队再批量调金蝶），吞吐会提升很多

### 5.3 个人成长收获

这次压测让我对外部系统依赖对整体性能的影响有了更直观的认识：

1. **性能优化不能靠猜：** 压测前以为瓶颈在数据库查询或业务逻辑，实际压测后发现金蝶 API 才是大头。没有压测数据支撑的优化都是瞎猜
2. **并发问题需要场景化分析：** 早期用同一采购订单做并发触发金蝶网控，后来改成不同订单才测出真实性能。压测数据构造必须符合真实业务场景
3. **监控和压测同等重要：** 只看 JMeter 的响应时间只能知道"慢"，不知道"为什么慢"。后续需要补齐连接池、线程池、外部调用链路的监控
4. **默认配置是坑：** Druid 默认连接数 8、线程池默认最大 25，这些默认值在低并发时没问题，一旦并发上来就会成为瓶颈。生产环境必须根据预期并发调整配置
