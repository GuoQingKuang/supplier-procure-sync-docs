# ProcureSync 压测指南

## 目录结构

```
load_test/
├── README.md                      # 本文件
├── generate_test_data.py          # 数据构造脚本
├── procure_sync_load_test.jmx     # JMeter 测试计划
└── load_test_data.csv             # 生成的压测数据（运行脚本后生成）
```

## 一、快速开始

### 1. 生成压测数据

**方式 A：模拟数据（推荐先调试 JMeter 流程）**

```bash
cd load_test
pip install pymssql pymysql  # 如果需要连接数据库
python generate_test_data.py --mock --count 350
```

生成 350 条模拟数据到 `load_test_data.csv`，这些数据不会通过业务校验，但可以验证 JMeter 请求流程。

**方式 B：真实数据（需要数据库访问）**

修改 `generate_test_data.py` 中的数据库连接配置后运行：

```bash
python generate_test_data.py --count 350
```

### 2. 运行 JMeter

1. 下载 JMeter：https://jmeter.apache.org/download_jmeter.cgi
2. 安装 JSON 插件（JMeter 5.x 内置，无需额外安装）
3. 打开 `procure_sync_load_test.jmx`
4. **修改用户变量**（测试计划层级）：
   - `BASE_URL`：你的服务器地址，如 `http://192.168.1.100:8080`
   - `SUPPLIER_PHONE`：测试供应商手机号
   - `SUPPLIER_PASSWORD`：登录密码
5. **运行方式**：

**GUI 模式（调试用，不建议压测）**：
```bash
jmeter -n -t procure_sync_load_test.jmx
```

**命令行模式（推荐）**：
```bash
# 300 并发发货接口压测
jmeter -n -t procure_sync_load_test.jmx \
  -Jthreads=300 -Jrampup=60 \
  -l results.jtl -e -o ./report/

# 100 并发列表查询压测（需先在 jmx 中启用对应线程组）
jmeter -n -t procure_sync_load_test.jmx \
  -Jthreads=100 -Jrampup=30 \
  -l list_results.jtl -e -o ./list_report/
```

6. 压测完成后查看 `./report/` 下的 HTML 报告

## 二、CSV 数据格式说明

### 单物料模式（默认）

每行一个发货物料项，JMeter 每次请求发送一个物料：

```csv
purchaseOrderNo,entryId,materialCode,materialVersion,materialName,materialSpec,deliveryQty,purchaseOrgId,purchaseOrgName,requestName
CGDD001,1001,WL001,A,轴承,6205-2RS,10,1,华东采购组织,submit_CGDD001_1001
```

JMeter 中构建的请求体：
```json
{
  "items": [{
    "purchaseOrderNo": "${purchaseOrderNo}",
    "entryId": ${entryId},
    "materialCode": "${materialCode}",
    "materialVersion": "${materialVersion}",
    "materialName": "${materialName}",
    "materialSpec": "${materialSpec}",
    "deliveryQty": ${deliveryQty},
    "purchaseOrgId": ${purchaseOrgId},
    "purchaseOrgName": "${purchaseOrgName}"
  }]
}
```

### 多物料合并模式

每 N 个物料合并为一个请求（需要同一采购组织）：

```bash
python generate_test_data.py --merge 3 --count 350
```

输出 CSV 格式：
```csv
requestName,items_json
submit_batch_0,[{"purchaseOrderNo":"CGDD001","entryId":1001,...}, ...]
```

这种模式需要在 JMeter 中修改请求体为 `{"items":${items_json}}`。

## 三、JMeter 测试计划说明

### 线程组

| 线程组 | 默认并发数 | Ramp-Up | 说明 |
|--------|-----------|---------|------|
| 提交发货 | 300 | 60秒 | **核心压测**，默认启用 |
| 送料通知单列表 | 100 | 30秒 | 查询压测，默认禁用 |
| 采购订单列表 | 100 | 30秒 | 查询压测，默认禁用 |

### 请求流程

```
1. POST /auth/login → 提取 token
2. POST /delivery-notice/submit → 用 token + CSV 数据提交发货
```

### 关键配置

- **CSV Data Set Config**：`shareMode=all`，所有线程共享数据池，每个线程取不同行
- **recycle=true**：数据用完后可循环（如果循环次数 > 1）
- **响应超时**：submit 接口设置 60 秒（金蝶 API 可能较慢）
- **响应断言**：验证 `"code":0` 判断成功

## 四、压测场景建议

### 场景 1：接口性能基线
- 用模拟数据（`--mock`），先跑通 JMeter 流程
- 验证 JMeter 配置、鉴权流程、响应断言都正常

### 场景 2：正常业务性能
- 用真实数据，300 并发，每个物料独立请求
- 观察：响应时间 P95/P99、QPS、错误率

### 场景 3：并发超发验证
- 用真实数据，但 CSV 中**让多个线程操作同一个 entryId**
- 观察：是否出现超发（发货总量 > 剩余数量）

### 场景 4：金蝶 API 超时影响
- 在测试环境模拟金蝶接口慢（如加 5 秒延迟）
- 观察：线程阻塞情况、连接池耗尽时间点

### 场景 5：阶梯加压
```bash
# 50 并发
jmeter -n -t procure_sync_load_test.jmx -Jthreads=50 -Jrampup=10 -l r50.jtl
# 100 并发
jmeter -n -t procure_sync_load_test.jmx -Jthreads=100 -Jrampup=20 -l r100.jtl
# 200 并发
jmeter -n -t procure_sync_load_test.jmx -Jthreads=200 -Jrampup=40 -l r200.jtl
# 300 并发
jmeter -n -t procure_sync_load_test.jmx -Jthreads=300 -Jrampup=60 -l r300.jtl
```

## 五、监控建议

压测时同时监控以下指标：

| 指标 | 工具 | 关注点 |
|------|------|--------|
| JVM 内存/GC | jstat, VisualVM | Full GC 频率、堆使用率 |
| 数据库连接池 | Druid Monitor | 活跃连接数、等待队列 |
| SQL 执行时间 | 慢查询日志 | 超过 1 秒的 SQL |
| 金蝶 API 调用 | 应用日志 | 超时次数、平均响应时间 |
| Redis 连接 | Redis INFO | 连接数、内存使用 |
| 系统 CPU/内存 | top, htop | CPU 使用率 > 80% 拐点 |

## 六、注意事项

1. **压测环境**：不要在生产数据库上压测，会生成大量脏数据（送货通知单）
2. **数据清理**：每次压测后需要清理 `delivery_notice` 表中的测试数据
3. **金蝶影响**：submit 接口会真实调用金蝶 API，确认测试环境金蝶是隔离的
4. **日志级别**：压测时建议将日志级别调整为 WARN，避免日志 IO 成为瓶颈
5. **查看结果树**：调试时启用，正式压测时**必须禁用**（非常耗内存）
6. **Token 管理**：当前设计每个线程先登录再调业务接口，300 并发登录本身也是一个压测点
