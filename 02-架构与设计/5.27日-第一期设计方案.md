# 供应商物料系统方案

## 一、项目背景

### 1.1 现状痛点

目前采购订单数据在金蝶 K3 Cloud ERP 系统中管理，供应商无法直接查看自己的订单状态、发货进度和验收结果。收料通知单的创建依赖人工在金蝶系统中操作，流程不透明、效率低。

### 1.2 项目目标

构建一个面向供应商的 Web 端系统，实现：
- 供应商可查看属于自己的采购订单及物料明细
- 供应商可对订单物料进行分批发货
- 发货后自动在金蝶系统中创建对应收料通知单
- 供应商可实时查看金蝶系统中的验收结果（二期，调通主流程后再做）

## 四、业务流程

### 4.1 完整链路流程

```
┌────────────────────── 管理端链路 ──────────────────────┐
│                                                        │
│  企业微信 OAuth2 登录                                   │
│       ↓                                                │
│  获取管理员身份（JWT token，携带角色标识）                │
│       ↓                                                │
│  查看金蝶供应商列表（调金蝶 API）                         │
│       ↓                                                │
│  选择供应商 → 设置账号密码 → 绑定金蝶供应商 ID            │
│       ↓                                                │
│  创建成功（写入 supplier_account 表）                     │
│                                                        │
└────────────────────────────────────────────────────────┘

┌────────────────────── 供应商端链路 ─────────────────────┐
│                                                        │
│  供应商账号密码登录                                     │
│       ↓                                                │
│  获取 JWT token（携带 supplierId）                       │
│       ↓                                                │
│  查看订单列表（查金蝶 SQL Server，按 supplierId + 状态过滤）     │
│  列表直接展示全部订单的物料明细行                         │
│       ↓                                                │
│  选择物料行（可跨订单选择） → 填写发货数量 → 点击发货      │
│       ↓                                                │
│  提交发货（POST /delivery-notice/submit，支持跨订单多物料）    │
│       ↓                                                │
│  后端校验发货数量 ≤ 剩余收货数量                          │
│       ↓                                                │
│  按采购订单分组，每组写入本地 delivery_notice 表           │
│       ↓                                                │
│  调用金蝶下推 API → 按订单生成收料通知单                   │
│       ↓                                                │
│  返回金蝶收料单唯一标识，更新到本地数据库                  │
│       ↓                                                │
│  调用金蝶保存 API → 修改收料数量                          │
│       ↓                                                │
│  成功：更新本地状态 = 2（已发货）                              │
│  失败：更新本地状态 = 3（已作废）                              │
│       ↓                                                │
│  前端刷新列表，显示新增送料记录                          │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 4.2 管理员创建供应商账号流程

```
企业微信 OAuth2 登录 → 获取管理员身份 → 查看金蝶供应商列表 → 选择供应商 → 设置供应商账号密码 → 绑定金蝶供应商ID → 创建成功
```

### 4.3 供应商发货流程

```
供应商登录 → 查看订单列表（全部订单物料行平铺展示）
    ↓
选择物料行（可跨订单选择） → 填写发货数量 → 勾选 → 点击发货
    ↓
后端校验:
    批量查金蝶 SQL Server（ErpMapper.selectBatchRemainQty），获取每个物料的"剩余收货数量"
    校验: 本次发货数量 ≤ 剩余收货数量
    ↓
步骤1: 所有物料分录合并为一次金蝶 Push 调用（支持跨订单），生成一张收料通知单
      写入本地 delivery_notice 表（存原始输入信息，status=1 发货中）
      生成通知单号 SN+yyyyMMdd+序号（Redis INCR + 降级方案）
    ↓
步骤2: 下推成功 → 金蝶返回收料单内码和编号 → 更新到本地数据库
    ↓
步骤3: 调用金蝶 Save API（批量修改收料单数量为本次发货数量）
    ↓
全部成功 → 更新本地状态=2（已发货），返回所有送料单信息
部分失败 → 标记为 3（已作废）
保存失败 → 调用金蝶作废 API 作废收料单，更新本地状态=3（已作废）或 4（异常），返回失败提示
作废也失败(极端) → 记录异常日志，后台人工处理
```

### 4.4 页面导航链路

```
订单列表页：直接展示全部订单的物料明细行，支持按单据编码、物料编码模糊查询和日期范围筛选
送料通知单页：展示全部送料通知单记录（成功记录查金蝶，失败记录查本地），支持按供应商/时间/状态过滤
面包屑导航: 首页 / 采购订单 / 送料通知单
```

### 4.5 发货功能交互流程

```
1. 供应商登录后查看订单列表，列表展示全部订单的物料行（平铺）
2. 在发货数量列输入发货数量（支持手动输入、全选一键填充）
3. 输入后自动勾选该行（校验：发货数量 ≤ 剩余收货数量）
4. 勾选后"发货"按钮可用，点击弹出确认弹窗
5. 确认弹窗展示选中的物料清单及发货数量
6. 确认后提交发货，所有分录合并为一次下推调用，生成一张收料通知单
7. 发货成功后刷新列表，展示新增送料记录
```

### 4.6 验收结果查询流程（二期）

```
供应商选择已验收订单 → 查Redis缓存 → 命中则返回
    ↓ 未命中
实时查询金蝶SQL Server → 写入Redis (TTL 1分钟) → 返回结果
```

## 七、金蝶对接方案

### 7.1 采购订单查询

| 方式 | 说明 |
|------|------|
| 优先: SQL 方式 | 直接查询金蝶 SQL Server 数据库 `T_PUR_POORDER` 表。按供应商 ID 和订单状态过滤，指定需要的字段返回 |
| 备选: API 方式 | 调用 `executeBillQueryJson`，FormId: `PUR_PurchaseOrder`，FilterString: `FSupplierId = {supplierId} AND FDOCUMENTSTATUS = 'C'`（仅正常状态）。用于 SQL 不可用时的兜底查询 |

### 7.2 创建收料通知单（下推方式）

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 调用**采购订单**下推 API | 所有物料分录合并为一次 Push 调用（支持跨订单），金蝶按收货方和供应商自动合并，生成一张收料通知单 |
| 2 | 调用**收料通知单**保存 API | 传入收料单内码，批量修改收料数量为本次发货数量 |

### 7.3 验收结果查询（二期）

| 方式 | 说明 |
|------|------|
| SQL 查询 | 实时查询金蝶 SQL Server 数据库，通过收料通知单关联采购入库单，获取合格数量等信息 |
| 缓存策略 | Redis 缓存，TTL 1 分钟，减少金蝶 DB 查询频率 |
| 具体表名和字段 | 待确认 |

### 7.4 关键关联关系

| 供应商账号 ↔ 金蝶供应商 | `supplier_account.supplier_id = T_BD_SUPPLIER.FSUPPLIERID` |
| 采购订单 ↔ 供应商 | `T_PUR_POORDER.FSUPPLIERID = supplier_account.supplier_id` |
| 收料通知单 ↔ 采购订单 | `T_PUR_RECEIVE.FSRCID = T_PUR_POORDER.FID` (源单内码) |
| 收料通知单分录 ↔ 采购订单分录 | `T_PUR_RECEIVEENTRY.FSRCENTRYID = T_PUR_POORDERENTRY.FENTRYID` |

## 八、异常处理与容错

### 8.1 跨系统异常处理（作废机制）

| 场景 | 处理方式 |
|------|----------|
| 下推API调用失败 | 本地送料单 status 标记为 3（已作废），记录失败原因，前端弹窗提示"系统繁忙，请稍后重试" |
| 保存API调用失败 | 1. 调用金蝶作废 API 作废已下推的收料单<br>2. 本地送料单 status 标记为 3（已作废），记录失败原因<br>3. 前端弹窗提示"系统繁忙，请稍后重试"<br>4. 若作废也失败（极端情况），记录异常日志，后台人工处理 |
| 网络超时（状态不确定） | 记录异常日志，后台人工核查金蝶侧状态并处理 |

### 8.2 发货数量校验

发货前批量查询金蝶 SQL Server（`ErpMapper.selectBatchRemainQty`），获取该物料的"需求数量"和"剩余收货数量"，校验本次发货数量 ≤ 剩余收货数量。以金蝶侧实时数据为准，避免本地累计数据与金蝶不一致导致超发。

## 十、需与需求方确认事项

以下事项待与需求方确认后补充到方案中：

### 10.1 已确认事项

1. **订单列表展示字段**（已确认）：
   - 单据类型、采购日期、单据编码、供应商、物料编码、物料版本、物料名称、规格、采购数量、剩余收料数量

3. **数据依赖方式**（已确认）：订单列表和送料通知单列表优先通过金蝶 SQL Server 查询，API 调用作为备选方案
