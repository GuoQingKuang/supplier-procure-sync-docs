# 收料单拆分需求 — 技术评估报告

## 一、需求概要

供应商生成预发货单后，可**按需点击"按物料拆分"**，将一张预发货单按 `materialCode` 拆分成多张子预发货单。每张子单独立转正，各自生成一张金蝶收料单。供应商可通过合并打印功能一次性打印所有子单。

**约束**：
- `delivery_notice` 表不改动，保持 1条 = 1张收料单
- `DeliveryNoticeService` 核心链路（`submit()`、`submitHeader()`）不动
- 部分失败时成功的正常流转，失败的单独重试

---

## 二、现有架构分析

### 当前数据模型

```
pre_delivery_notice (1条) ──转正──> delivery_notice (1条) ──1:1──> kingdee_receive_bill_id (1个)
```

### 当前流程链路

```
预发货草稿 → 编辑表头/库位 → submitToFormal() → submit() → Push(1次) + Save(1次) → delivery_notice → 收料单
```

核心假设：**一次转正对应一张金蝶收料单**。这个假设贯穿 `submit()`、`detail()`、`list()`、`submitHeader()` 所有方法。

### 业务流程顺序

```
生成预发货单 → 打印 → 实际送货 → 回填库位 → 正式发货（调金蝶接口）
```

注意：送货和系统发货是分离的，供应商先送货后操作系统，正常场景下正式发货不应失败。

---

## 三、方案选型

### 方案A：转正时自动拆分

```
submitToFormal() → 按物料编码分组 → 创建M张子单 → 批量转正
```

**问题**：
- 拆分发生在转正时，供应商无法在拆分后编辑各子单的库位信息
- 批量转正耗时较长，需要异步化处理
- 合并打印需要额外保留母单快照

### 方案B：生成预发货单后手动拆分（推荐）

```
供应商点击"按物料拆分" → 创建M张子预发货单 → 子单各自编辑/打印/转正
```

**优点**：
- 每张子单 1:1 对应金蝶收料单，现有 `submit()` 逻辑完全复用
- 拆分后供应商可逐张编辑子单的库位/备注
- 失败影响范围限定在单张子单
- `delivery_notice` 表不需要改
- `DeliveryNoticeService` 完全不动

**缺点**：
- 需要新增拆分接口 + 子单关联关系字段
- 供应商需要多一点操作（但提供批量发货/打印功能）

### 选型结论：方案B

改动集中在预发货管理层，`DeliveryNoticeService` 核心链路完全不动。

## 四、数据库变更

### `pre_delivery_notice` 表新增字段

```sql
ALTER TABLE pre_delivery_notice ADD COLUMN parent_notice_no VARCHAR(32) COMMENT '母预发货单号（拆分来源）';
ALTER TABLE pre_delivery_notice ADD INDEX idx_parent_notice_no (parent_notice_no);
```

**说明**：
- 不需要 `is_print_only` 字段，母单拆分后标记为"已拆分"状态即可
- 不需要 `split_status` 字段，用现有 `status` + `parent_notice_no` 即可表达

### `delivery_notice` 表

**不需要改动**。保持1条记录对应1个 `kingdeeReceiveBillId`。

## 五、核心风险与技术细节

### 5.1 拆分逻辑

**实现方式**：

```
1. 校验母单状态为草稿，未被拆分过
2. 读取母单 materialData JSON，按 materialCode 分组
3. 为每组创建一张子 pre_delivery_notice：
   - 继承母单的 supplierId/purchaseOrgId/warehouseGroup/pendingArea/remark
   - materialData 仅包含该组的物料行
   - parent_notice_no = 母单 noticeNo
   - status = 0（草稿）
4. 母单 status 标记为"已拆分"（可用特殊状态值或 parent_notice_no IS NOT NULL 判断）
5. 返回子单列表
```

**并发控制**：拆分时对母单加乐观锁 `UPDATE ... WHERE id = ? AND status = 0 AND parent_notice_no IS NULL`，防止重复拆分。

### 5.2 批量发货

**问题**：N 张子单串行调用 `submitToFormal()` = 2N 次金蝶 HTTP 调用。假设 10 张子单，约 20-40 秒。

**解决**：

- **方案A（推荐）**：批量发改用异步。后端提交任务到线程池，立即返回任务ID。前端轮询进度接口查看成功/失败结果。
- **方案B**：前端分批调用，每次选5张，用户自己控制节奏。

**异步实现要点**：
```java
@Async
public BatchShipResult batchShip(List<String> noticeNos) {
    // 串行循环调用 submitToFormal()
    // 加调用间隔（500ms）避免金蝶限流
    // 记录每张成功/失败状态
    // 返回汇总结果
}
```

### 5.3 并发重复提交

**问题**：供应商点击发货后网络超时，重复点击 → 同一子单被转正两次。

**解决**：
- `submitToFormal()` 已有状态校验（仅草稿可转正），复用即可
- 额外加 Redis 分布式锁：`lock:pre_delivery:{noticeNo}`，超时30s

### 5.4 金蝶接口稳定性

**问题**：批量调用中失败原因分两种：
- **临时性**：网络抖动、超时 → 重试可解决
- **业务性**：物料被冻结、基础资料无效 → 重试无效

**应对**：
- 区分错误类型，业务性错误直接标记失败，不自动重试
- 提供手动重试按钮，供应商可自行决定重试次数
- 记录每次调用的请求/响应 JSON 到 `kingdee_push_params` / `fail_reason`

### 5.5 合并打印

**实现方式**：
- 接收多个子单 noticeNo 列表
- 查询各子单的 `materialData`，合并所有物料行
- 生成统一格式的 PDF（可按子单分组，每组带子单号）
- 不涉及金蝶接口，纯本地数据

### 5.6 母单拆分后的处理

拆分后母单标记为"已拆分"，禁止以下操作：
- 编辑（表头/物料行）
- 转正发货
- 删除

母单仅用于：
- 展示/追溯拆分来源
- 合并打印时可从母单读取原始30行物料（或从子单合并）

---

## 七、不可控因素

| 因素 | 说明 |
|------|------|
| 金蝶接口限流 | 短时间内密集调用可能触发限流，批量发货需加调用间隔 |
| 金蝶 Push 重复下推 | 同一条 entryId 重复 Push 会怎样（报错还是重复扣减），上线前需验证 |
| 跨采购订单下推 | 同物料编码可能来自不同采购订单，金蝶是否支持一次性下推需验证 |

---

## 八、结论

1. 推荐**方案B（手动拆分）**，改动集中在预发货管理层，`DeliveryNoticeService` 完全不动
2. 核心风险是**批量发货耗时**需要异步化，以及**金蝶接口限流**需要控制调用间隔
3. 数据库仅需新增1个字段，`delivery_notice` 表不需要改
4. 开发周期约 6-8 天，主要工作量在批量操作功能和前端交互
5. 上线前需验证：金蝶 Push 接口对同一条 entryId 重复下推的行为
