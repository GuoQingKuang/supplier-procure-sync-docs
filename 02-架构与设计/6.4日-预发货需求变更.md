# 需求变更记录

**提出日期：** 2026-06-04
**提出方：** 需求方
**状态：** 开发完成，待上线验证

---

## 一、变更原因

需求方在金蝶侧收到送料通知单后，需要手动在金蝶系统填写库房分组、待检区仓位和备注，希望改为在供应商系统中完成填写，减少金蝶侧的手工操作。

**说明：**
- 库房分组、待检区仓位、备注属于**送料通知单表头级别**（整单共用），填写后需同步到金蝶收料通知单表头中。
- 物料行级别的库位（location）和备注（remark）字段仍保留在预发货单中，供应商可按物料行填写/修改，**仅本地存储**，不传给金蝶。

---

## 二、变更内容

### 2.1 原有流程（不变）

> 供应商勾选物料 → 填写发货数量 → 提交发货 → 调用金蝶Push/Save/Audit → 生成收料通知单（审核中） → 金蝶侧填写库房分组/仓位/备注

### 2.2 变更后流程（预发货流程）

**阶段一：预发货（草稿态，status=0）**
供应商勾选物料 → 填写发货数量 → 【预发货】 → 本地生成预发货通知单（草稿态，含库房分组/待检区仓位/备注字段，可留空后补）

**阶段二：预发货编辑**
供应商打印预发货通知单 → 按单备货送货 → 从金蝶/仓库拿到实际的库房分组、待检区仓位、备注信息 → 在预发货通知单表头填写/修改这三个字段；同时可按物料行填写/修改库位、备注（仅本地存储）
- **草稿态**可编辑：表头字段 + 物料行（库位/备注/发货数量）
- **已转正态**可编辑：仅表头字段

**阶段三：正式发货（转正，status=1）**
供应商点击【正式发货】 → 校验金蝶剩余数量 → 调用金蝶Push（下推收料通知单）→ Save（只改数量 FActReceiveQty）→ 成功则本地生成 delivery_notice（status=已发货）→ 预发货单标记为已转正
此时金蝶单据状态 = **提交**

**阶段四：表头字段填写 + 提交（status=3，已提交）**
供应商确认表头三个字段正确 → 点击【提交】→ Save（只更新三个表头字段：库房分组/待检区仓位/备注）→ Submit（改状态为审核中）→ 预发货单状态改为"已提交"
如提交失败（如库房分组不存在），返回错误提示，用户重新填写后再次提交，可多次尝试

**阶段五：已提交态（status=3）**
提交成功后预发货单显示"已提交"状态，用户可再次编辑表头字段或打印

### 2.3 具体变更项

| 序号 | 变更项 | 详细说明 |
|:----:|--------|----------|
| 1 | 新增"预发货"按钮 | 采购订单列表页，与现有"发货"按钮并列 |
| 2 | 预发货通知单管理 | 新增预发货通知单列表页，支持查看、编辑、打印、正式发货、提交 |
| 3 | 预发货本地入库 | 调用预发货接口时不调用金蝶任何接口，仅在本地数据库新建 pre_delivery_notice 记录（status=0 草稿） |
| 4 | 表头编辑能力 | 预发货通知单表头可编辑：库房分组、待检区仓位、备注（整单级别，需同步到金蝶）。草稿态和已转正态均可编辑表头 |
| 5 | 物料行编辑能力 | 仅草稿态可编辑物料行：库位（location）、备注（remark）、发货数量（仅本地存储，不传给金蝶） |
| 6 | 预发货打印 | 预发货通知单支持打印，打印模板表头展示库房分组/待检区仓位/备注，标注"预发货"标识 |
| 7 | 正式发货接口 | 从预发货单触发正式发货，读取 pre_delivery_notice 物料明细，调用金蝶Push/Save（只改数量）流程，生成 delivery_notice 正式单 |
| 8 | 金蝶 Save 接口改造（阶段三） | Save API 调用时只传 FActReceiveQty，不传表头字段 |
| 9 | 提交表头 + 提交接口（阶段四） | 新增独立接口：从预发货单读取表头字段 → Save（F_RWBQ_Combo1/F_RWBQ_Text8/FNote）+ Submit（改状态为审核中），两步合并，失败可重试 |
| 10 | 新建预发货表 | 新建 pre_delivery_notice 表，与 delivery_notice 物理隔离，职责独立 |
| 11 | 正式发货时数量校验 | 正式发货时重新查询金蝶剩余可发数量，防止预发货到正式发货期间数量被占用 |
| 12 | 送料通知单列表 | 列表查询包含 status=5（审核中）的记录，提交成功后送料通知单仍在列表中可见 |

## 四、技术要点与决策

### 4.1 金蝶字段映射

| 业务字段 | 金蝶二开字段名 | 说明 |
|----------|---------------|------|
| 库房分组 | `F_RWBQ_Combo1` | 下拉框类型，输入不匹配则 Save 静默忽略，Submit 时报"必录"错误 |
| 待检区仓位 | `F_RWBQ_Text8` | 文本输入框类型，正常保存 |
| 备注 | `FNote` | 标准字段，正常保存 |

### 4.2 状态流转

```
预发货单：草稿(0) → [正式发货] → 已转正(1) → [提交表头+Submit] → 已提交(3)
                                                          ↓（可选）
                                                      已作废(4)

送料通知单：发货中(1) → [金蝶Save+Submit] → 审核中(5)
```

### 4.3 submitHeader 数据来源

提交表头时，三个字段（库房分组/待检区仓位/备注）**从预发货单读取**，而非从 delivery_notice 表读取。因为转正时 delivery_notice 中这三个字段为 null，用户填写的值始终在 pre_delivery_notice 表中。

### 4.4 错误处理

- **库房分组不匹配**：Save 时金蝶静默忽略，Submit 时报"必录"错误，前端提示"库房分组名称不正确，请检查输入是否在系统可选范围内"
- **金蝶 Submit 失败**：不改变送料通知单状态（保持"已发货"），允许用户重试
- **预发货单编辑**：仅草稿态和已转正态可编辑，已提交态和已作废态不可编辑

## 五、连带影响

1. **需求1（审核中）已合并到此方案中**：阶段四 Save（表头字段）成功后追加 Submit 调用，无需单独排期
2. **需求2（发货后改数量）被此方案替代**：供应商在预发货阶段可修改数量，正式发货后不可修改，无需额外开发
3. **现有发货流程保留**：直接发货（不经过预发货）的流程保持不变，两种方式并存
4. **表头与物料行区分**：库房分组/待检区仓位/备注为表头级别（同步金蝶），物料行的库位/备注仅本地存储
5. **责任边界**：库房分组填写错误导致提交失败由用户自行联系金蝶处理，开发侧提供自定义错误提示
