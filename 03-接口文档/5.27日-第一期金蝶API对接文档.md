### 3.2 核心表关联关系

| 关联 | 条件 |
|------|------|
| 供应商账号 ↔ 金蝶供应商 | `supplier_account.supplier_id = T_BD_SUPPLIER.FSUPPLIERID` |
| 采购订单 ↔ 供应商 | `T_PUR_POORDER.FSUPPLIERID = supplier_account.supplier_id` |
| 收料通知单 ↔ 采购订单 | `T_PUR_RECEIVE.FSRCID = T_PUR_POORDER.FID`（源单内码） |
| 收料通知单分录 ↔ 采购订单分录 | `T_PUR_RECEIVEENTRY.FSRCENTRYID = T_PUR_POORDERENTRY.FENTRYID`（源单分录内码） |
| 收料通知单分录内码（收料单侧） | Push 返回的 `EntryIds.FDetailEntity[]` |

**跨系统数据追溯链路**：
```
采购订单分录内码(FEntryID)
    → Push API EntryIds 参数
    → 生成收料通知单(FID) + 收料单分录内码数组(EntryIds.FDetailEntity)
    → 收料单分录.FSRCENTRYID = 原始采购订单分录内码（可追溯）
    → 本地 delivery_notice 表 kingdeeReceiveBillId = 收料单FID
```

#### 4.1 下推接口（Push）：采购订单 → 收料通知单

**测试类**：`KingdeePushTest.java`

**请求参数**：
```json
{
    "Ids": "",
    "Numbers": [],
    "EntryIds": "267044,280882,287705,287709",
    "RuleId": "PUR_PurchaseOrder-PUR_ReceiveBill",
    "TargetBillTypeId": "",
    "TargetOrgId": 0,
    "TargetFormId": "",
    "IsEnableDefaultRule": false,
    "IsDraftWhenSaveFail": false,
    "CustomParams": {}
}
```

**调用方式**：`client.push("PUR_PurchaseOrder", pushJson)`

**成功响应**：
```json
{
    "Result": {
        "ResponseStatus": {
            "IsSuccess": true,
            "Errors": [],
            "SuccessEntitys": [{
                "Id": 250451,
                "Number": "CGSLBOCI20260525002",
                "DIndex": 0,
                "EntryIds": { "FDetailEntity": [360284, 360285, 360286, 360287] }
            }],
            "MsgCode": 0
        },
        "ConvertResponseStatus": { "IsSuccess": true, "MsgCode": 0 }
    }
}
```

**关键返回字段**：
- `Result.ResponseStatus.SuccessEntitys[0].Id` → 收料通知单内码（即 `kingdeeReceiveBillId`）
- `Result.ResponseStatus.SuccessEntitys[0].Number` → 收料通知单编号
- `Result.ResponseStatus.SuccessEntitys[0].EntryIds.FDetailEntity[]` → 收料单分录内码数组（用于后续 Save 修改数量）

**EntryIds 规则**：
- EntryIds（采购订单分录内码）在金蝶系统中**全局唯一**
- 多个分录内码用逗号分隔，如 `"267044,280882,287705,287709"`
- 金蝶按**收货方和供应商自动分组**，同供应商同收货方的分录自动合并为**一张**收料通知单
- 支持跨订单多分录下推，一次 Push 调用可跨多个采购订单

**多订单下推测试结果**（4条分录来自3个不同采购订单）：

| 分录内码 | 源订单编号 | 物料编码 | 物料名称 | 剩余收料数量 |
|---------|-----------|---------|---------|------------|
| 267044 | BCBOCI251210012 | 9540580 | 门板搭扣 | 5 |
| 280882 | BCBOCI260316031 | 8457381 | 翻转镜座 | 1 |
| 287705 | BCBOCI260413049 | 7214347 | 下二保护镜前面板 | 9 |
| 287709 | BCBOCI260413049 | 8457381 | 翻转镜座 | 6 |

**结果**：4 条分录（3个不同订单）**合并生成了 1 张收料通知单**（Id=250451），包含 4 条分录行。

**常见错误**：
- `TargetOrgId` 不能传空字符串 `""`，必须传整数 `0`
- `TargetFormId` 不需要传，留空字符串即可

**请求参数**：


**调用方式**：`client.view("PUR_ReceiveBill", viewJson)`

**关键字段映射**（View 返回的 JSON key 与 Save API 文档的差异）：

| Save API 文档字段名 | View 返回 JSON key | 说明 |
|---------------------|-------------------|------|
| FBillNo | BillNo | 单据编号 |
| FDocumentStatus | DocumentStatus | 单据状态 |
| FDate | Date | 收料日期 |
| FStockOrgId | StockOrgId | 收料组织（复合对象，含 Number 字段） |
| FSupplierId | SupplierId | 供应商（复合对象，含 Number 字段） |
| FId | Id | 收料单内码（单据头） |
| FDetailEntity | PUR_ReceiveEntry | 分录数组 key |
| FEntryID | Id | 分录内码（注意：分录行的内码 key 是 `Id`，不是 `FEntryID`） |
| FMaterialId | MaterialID | 物料编码（复合对象，含 Number 字段） |
| FActReceiveQty | ActReceiveQty | 交货数量 |
| FActlandQty | ActlandQty | 实收数量 |
| FMustQty | MustQty | 应收数量 |
| FOrderBillNo | OrderBillNo | 源单单号 |
| FSrcId | SrcId | 源单内码 |
| FSrcEntryId | SrcEntryId | 源单分录内码 |
| FSUPDELQTY | SupDelQty | 供应商送货数量 |
| F_RWBQ_Text1 | F_RWBQ_Text1 | 物料版本（自定义字段，保留完整名称） |
| F_RWBQ_Text6 | F_RWBQ_Text6 | 物料编码文本（自定义） |
| F_Name_Text | F_Name_Text | 物料名称文本（自定义） |

**View 返回规律**：
- 标准字段去掉 `F` 前缀（如 `FMaterialId` → `MaterialID`）
- 自定义字段保留完整名称（如 `F_RWBQ_Text1` 不变）
- 分录数组 key 是 `PUR_ReceiveEntry`，不是 `FDetailEntity`
- 分录行内码 key 是 `Id`，不是 `FEntryID`

**注意**：View 接口返回数据量很大（每条分录 100+ 字段），**供应商端列表推荐使用 SQL Server 查询**，避免冗余数据传输。

#### 4.3 保存/修改接口（Save）：修改收料通知单数量

**测试类**：`KingdeeSaveTest.java`

**单条分录修改测试**：收料单 Id=250450，分录 Id=360283，交货数量从 15 改为 5

**最小化请求参数**（经验证可用）：
```json
{
    "NeedUpDateFields": ["FActReceiveQty"],
    "IsDeleteEntry": "false",
    "Model": {
        "FID": 250450,
        "FDetailEntity": [
            {
                "FEntryID": 360283,
                "FActReceiveQty": 5
            }
        ]
    }
}
```

**多条分录修改测试**：收料单 Id=250451，4 条分录都改为 1

```json
{
    "NeedUpDateFields": ["FActReceiveQty"],
    "IsDeleteEntry": "false",
    "Model": {
        "FID": 250451,
        "FDetailEntity": [
            { "FEntryID": 360284, "FActReceiveQty": 1 },
            { "FEntryID": 360285, "FActReceiveQty": 1 },
            { "FEntryID": 360286, "FActReceiveQty": 1 },
            { "FEntryID": 360287, "FActReceiveQty": 1 }
        ]
    }
}
```

**调用方式**：`client.save("PUR_ReceiveBill", saveJson)`

**成功响应**：
```json
{
    "Result": {
        "ResponseStatus": {
            "IsSuccess": true,
            "Errors": [],
            "SuccessEntitys": [{"Id": 250451, "Number": "CGSLBOCI20260525002", "DIndex": 0}],
            "MsgCode": 0
        },
        "Id": 250451,
        "Number": "CGSLBOCI20260525002",
        "NeedReturnData": [{}]
    }
}
```

**关键规则**：
1. `NeedUpDateFields` 中的字段名**不带分录前缀**，直接写 `"FActReceiveQty"` 而非 `"FDetailEntity.FActReceiveQty"`
2. Model 中必须包含收料单内码 `FID`（单据头）和分录内码 `FEntryID`（分录行）
3. Save 接口传参字段名用 **F 前缀形式**（如 `FActReceiveQty`），与 View 返回的去 F 前缀形式不同
4. 修改 `FActReceiveQty` 后，金蝶自动联动更新 `FActlandQty`、`FMustQty`、`FStockInMaxQty`、`FStockInMinQty`、`FStockQty` 等关联字段，以及金额相关字段（`FAmount`、`FTaxAmount`、`FAllAmount` 等）
5. `IsDeleteEntry` 必须设为 `"false"`，避免未传入的字段被清空
6. 多条分录修改时，在 `FDetailEntity` 数组中依次添加即可，一次调用完成

**Save API 完整参数说明**（非必录参数可省略）：

| 参数 | 类型 | 必录 | 说明 |
|------|------|------|------|
| NeedUpDateFields | Array | 否 | 需要更新的字段数组，格式 `["FActReceiveQty"]` |
| NeedReturnFields | Array | 否 | 需返回结果的字段数组 |
| IsDeleteEntry | Boolean | 否 | 是否删除已存在的分录，默认 true，修改时必须设为 false |
| ValidateFlag | Boolean | 否 | 是否验证数据合法性，默认 true |
| NumberSearch | Boolean | 否 | 是否用编码搜索基础资料，默认 true |
| IsAutoAdjustField | Boolean | 否 | 是否自动调整 JSON 字段顺序，默认 false |
| Model | JSON | 是 | 表单数据包，修改时必须包含 FID 和 FDetailEntity 中的 FEntryID |

#### 4.6 接口调用完整流程

```
发货流程：
1. 写入本地 delivery_notice 表（存 kingdeeReceiveBillId，status=1 发货中，记录原始物料明细JSON）

2. 采购订单下推收料通知单：
    client.push("PUR_PurchaseOrder", pushJson)
    → 返回收料单内码(Id)和编号(Number) + 收料单分录内码数组(EntryIds.FDetailEntity)
    → 更新本地 delivery_notice 表 kingdee_receive_bill_id 和 kingdee_receive_bill_no

3. 修改收料通知单数量：
    client.save("PUR_ReceiveBill", saveJson)
    → 返回 IsSuccess=true
    → 更新本地 delivery_notice 表状态为"已发货"（status=2）

查询流程：
    收料通知单详情：优先查金蝶 SQL Server（T_PUR_RECEIVE + T_PUR_RECEIVEENTRY）
    → 按收料单 ID 查询单张详情
    → 按供应商 ID 查询全部送料记录
```

### 5. 收料通知单 SQL Server 查询已确认

**数据策略**：收料通知单列表和详情**优先通过 SQL Server 查询**，避免 View API 返回 100+ 冗余字段。

**核心表**：
- `T_PUR_RECEIVE`：收料通知单主表（单据头）
- `T_PUR_RECEIVEENTRY`：收料通知单分录表（单据体）
- `T_PUR_RECEIVEENTRY_R`：收料通知单分录扩展表（含确认交货数量等）
- `T_BD_MATERIAL`：物料表（物料编码）
- `T_BD_MATERIAL_L`：物料多语言表（物料名称）
- `T_BD_SUPPLIER_L`：供应商多语言表（供应商名称）

**SQL 1：按收料单 ID 查询（查看单张收料单物料明细）**
```sql
SELECT
     rec.FID 单据ID,
     rec.FBILLNO 单据编号,
     rec.FDOCUMENTSTATUS 单据状态,
     rec.FCANCELSTATUS 作废状态,
     rec.FCLOSESTATUS 整单关闭状态,
     rec.FSTOCKORGID 收料组织,
     rec.FDATE 收料日期,
     rec.FSUPPLIERID 供应商id,
     supl.FNAME 供应商名称,
     rec.FSENDBILLNO 送货单号,
     rec.FCREATEDATE 创建日期,

     rece.FENTRYID 明细内码,
     tbm.FNUMBER 物料编码,
     tbml.FNAME 物料名称,

     rece.FACTLANDQTY 实到数量,
     rece.FACTRECEIVEQTY 交货数量,
     rece.FMUSTQTY 应收数量,
     rece.FORDERBILLNO 源采购订单单号,
     rece.FSRCID 源单内码,
     rece.FSRCENTRYID 源单分录内码,
     rece.FSRCFORMID 源单类型,
     rece.FSRCBILLNO 源单单号,

     recer.FCONFIRMDELIQTY 确认交货数量,
     recer.FCONFIRMDELIDATE 确认到货日期,

     rece.F_RWBQ_Text1 物料版本,
     rece.F_Name_Text 物料名称自定义,
     rece.F_RWBQ_Text6 物料编码自定义,
     rece.F_RWBQ_Text7 备注

FROM T_PUR_RECEIVE rec
LEFT JOIN T_BD_SUPPLIER_L supl ON rec.FSUPPLIERID = supl.FSUPPLIERID AND supl.FLOCALEID = 2052
LEFT JOIN T_PUR_RECEIVEENTRY rece ON rec.FID = rece.FID
LEFT JOIN T_PUR_RECEIVEENTRY_R recer ON recer.FENTRYID = rece.FENTRYID
LEFT JOIN T_BD_MATERIAL tbm ON rece.FMATERIALID = tbm.FMATERIALID AND tbm.FUSEORGID = rec.FSTOCKORGID
LEFT JOIN T_BD_MATERIAL_L tbml ON rece.FMATERIALID = tbml.FMATERIALID AND tbml.FUSEORGID = rec.FSTOCKORGID AND tbml.FLOCALEID = 2052
WHERE rec.FID = ?
  AND rece.FSRCFORMID = 'PUR_PurchaseOrder'
```

**SQL 2：按供应商查询送料记录（供应商端送料通知单列表）**
```sql
SELECT
     rec.FID 单据ID,
     rec.FBILLNO 单据编号,
     rec.FDOCUMENTSTATUS 单据状态,
     rec.FDATE 收料日期,
     rec.FSUPPLIERID 供应商id,
     supl.FNAME 供应商名称,

     rece.FENTRYID 明细内码,
     tbm.FNUMBER 物料编码,
     tbml.FNAME 物料名称,

     rece.FACTLANDQTY 实到数量,
     rece.FACTRECEIVEQTY 交货数量,
     rece.FMUSTQTY 应收数量,
     rece.FORDERBILLNO 源采购订单单号,
     rece.FSRCENTRYID 源单分录内码,

     recer.FCONFIRMDELIQTY 确认交货数量,
     recer.FCONFIRMDELIDATE 确认到货日期,

     rece.F_RWBQ_Text1 物料版本,
     rece.F_RWBQ_Text7 备注

FROM T_PUR_RECEIVE rec
LEFT JOIN T_BD_SUPPLIER_L supl ON rec.FSUPPLIERID = supl.FSUPPLIERID AND supl.FLOCALEID = 2052
LEFT JOIN T_PUR_RECEIVEENTRY rece ON rec.FID = rece.FID
LEFT JOIN T_PUR_RECEIVEENTRY_R recer ON recer.FENTRYID = rece.FENTRYID
LEFT JOIN T_BD_MATERIAL tbm ON rece.FMATERIALID = tbm.FMATERIALID AND tbm.FUSEORGID = rec.FSTOCKORGID
LEFT JOIN T_BD_MATERIAL_L tbml ON rece.FMATERIALID = tbml.FMATERIALID AND tbml.FUSEORGID = rec.FSTOCKORGID AND tbml.FLOCALEID = 2052
WHERE rec.FSUPPLIERID = ?
  AND rece.FSRCFORMID = 'PUR_PurchaseOrder'
  AND rec.FDOCUMENTSTATUS = 'A'
ORDER BY rec.FDATE DESC
```
