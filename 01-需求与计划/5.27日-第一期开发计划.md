# 供应商物料系统 - 开发方案与计划

## 一、项目背景

### 1.1 现状痛点

目前采购订单数据在金蝶 K3 Cloud ERP 系统中管理，供应商无法直接查看自己的订单状态、发货进度和验收结果。收料通知单的创建依赖人工在金蝶系统中操作，流程不透明、效率低。

### 1.2 项目目标

构建一个面向供应商的 Web 端系统，实现：
- 供应商可查看属于自己的采购订单及物料明细
- 供应商可对订单物料进行分批发货
- 发货后自动在金蝶系统中创建对应收料通知单
- 供应商可实时查看金蝶系统中的验收结果（二期，调通主流程后再做）

## 四、数据库设计

### 4.1 本地 MySQL 数据库（procure_sync）

#### `supplier_account`（供应商账号表）

```sql
CREATE TABLE `supplier_account` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `supplier_id` BIGINT NOT NULL COMMENT '金蝶供应商内码(FSUPPLIERID)',
  `supplier_code` VARCHAR(64) NOT NULL COMMENT '金蝶供应商编码',
  `supplier_name` VARCHAR(256) NOT NULL COMMENT '供应商名称',
  `username` VARCHAR(64) NOT NULL COMMENT '登录用户名（唯一）',
  `password` VARCHAR(256) NOT NULL COMMENT '登录密码（明文）',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '0-禁用 1-启用',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `create_by` BIGINT DEFAULT NULL COMMENT '创建人(admin_user.id)',

  UNIQUE KEY `uk_username` (`username`),
  KEY `idx_supplier_id` (`supplier_id`)
```

```sql
CREATE TABLE `delivery_notice` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,

  `supplier_id` BIGINT NOT NULL COMMENT '供应商内码',

  `kingdee_update_params` JSON DEFAULT NULL COMMENT '金蝶修改API传参留样',
  `fail_reason` VARCHAR(512) DEFAULT NULL COMMENT '失败原因',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

  UNIQUE KEY `uk_notice_no` (`notice_no`),
  KEY `idx_supplier_id` (`supplier_id`),
  KEY `idx_ship_time` (`ship_time`),
  KEY `idx_status` (`status`)

```

### 6.2 采购订单查询（SQL Server）

```sql
SELECT
    pur.FID 单据ID,
    pur.FBILLTYPEID 单据类型,
    pur.FDATE 采购日期,
    pur.FSUPPLIERID 供应商,
    pur.FBILLNO 单据编号,
    pure.FENTRYID 分录内码,
    pure.FMATERIALID 物料编码,
    pure.F_RWBQ_Text5 物料版本,
    tbml.FNAME 物料名称,
    pure.FQTY 采购数量,
    purer.FREMAINRECEIVEQTY 剩余收料数量,
    pur.FDOCUMENTSTATUS 单据状态,
    pur.FCANCELSTATUS 作废状态,
    pur.FCLOSESTATUS 关闭状态,
    pure.FMRPCloseStatus 业务关闭状态,
    pure.FMRPFreezeStatus 业务冻结,
    pure.FMRPTerminateStatus 业务终止
FROM T_PUR_POORDER pur
LEFT JOIN T_BD_SUPPLIER_L supl ON pur.FSUPPLIERID = supl.FSUPPLIERID AND supl.FLOCALEID = 2052
LEFT JOIN T_PUR_POORDERENTRY pure ON pur.FID = pure.FID
LEFT JOIN T_PUR_POORDERENTRY_R purer ON pure.FENTRYID = purer.FENTRYID
LEFT JOIN T_PUR_POORDERENTRY_L purel ON pure.FENTRYID = purel.FENTRYID AND purel.FLOCALEID = 2052
LEFT JOIN T_BD_MATERIAL tbm ON pure.FMATERIALID = tbm.FMATERIALID AND tbm.FUSEORGID = pur.FPURCHASEORGID
LEFT JOIN T_BD_MATERIAL_L tbml ON pure.FMATERIALID = tbml.FMATERIALID AND tbml.FUSEORGID = pur.FPURCHASEORGID AND tbml.FLOCALEID = 2052
WHERE pur.FSUPPLIERID = ?
  AND pur.FDOCUMENTSTATUS = 'C'
  AND pur.FCLOSESTATUS = 'A'
  AND pur.FCANCELSTATUS = 'A'
  AND pure.FMRPFreezeStatus = 'A'
  AND pure.FMRPCloseStatus = 'A'
  AND pure.FMRPTerminateStatus = 'A'
  AND purer.FREMAINRECEIVEQTY > 0
```

### 6.4 金蝶 API 调用

#### Push（下推：采购订单 → 收料通知单）

```json
{
    "Ids": "",
    "Numbers": [],
    "EntryIds": "267044,280882,287705",
    "RuleId": "PUR_PurchaseOrder-PUR_ReceiveBill",
    "TargetOrgId": 0,
    "TargetFormId": "",
    "IsEnableDefaultRule": false,
    "IsDraftWhenSaveFail": false,
    "CustomParams": {}
}
```
调用：`client.push("PUR_PurchaseOrder", pushJson)`

**关键返回：**
- `Result.ResponseStatus.SuccessEntitys[0].Id` → 收料通知单内码
- `Result.ResponseStatus.SuccessEntitys[0].EntryIds.FDetailEntity[]` → 收料单分录内码数组

#### Save（修改收料通知单数量）

```json
{
    "NeedUpDateFields": ["FActReceiveQty"],
    "IsDeleteEntry": "false",
    "Model": {
        "FID": 250451,
        "FDetailEntity": [
            { "FEntryID": 360284, "FActReceiveQty": 1 }
        ]
    }
}
```
调用：`client.save("PUR_ReceiveBill", saveJson)`

#### Cancel（作废收料通知单）

```json
{
    "CreateOrgId": 0,
    "Numbers": [],
    "Ids": "250451",
    "PkEntryIds": [],
    "UseOrgId": 0,
    "NetworkCtrl": "",
    "IgnoreInterationFlag": ""
}
```
调用：`client.excuteOperation("PUR_ReceiveBill", "Cancel", cancelJson)`

### 6.7 核心表关联关系

| 关联 | 条件 |
|------|------|
| 供应商账号 ↔ 金蝶供应商 | `supplier_account.supplier_id = T_BD_SUPPLIER.FSUPPLIERID` |
| 采购订单 ↔ 供应商 | `T_PUR_POORDER.FSUPPLIERID = supplier_account.supplier_id` |
| 收料通知单 ↔ 采购订单 | `T_PUR_RECEIVE.FSRCID = T_PUR_POORDER.FID` |
| 收料通知单分录 ↔ 采购订单分录 | `T_PUR_RECEIVEENTRY.FSRCENTRYID = T_PUR_POORDERENTRY.FENTRYID` |

**跨系统数据追溯链路：**
```
采购订单分录内码(FEntryID)
    → Push API EntryIds 参数
    → 生成收料通知单(FID) + 收料单分录内码数组(EntryIds.FDetailEntity)
    → 收料单分录.FSRCENTRYID = 原始采购订单分录内码（可追溯）
    → 本地 delivery_notice 表 kingdeeReceiveBillId = 收料单FID
```

## 七、核心业务流程

### 7.1 供应商发货完整流程

```
供应商登录 → 查看订单列表（全部订单物料行平铺展示）
    ↓
选择物料行（可跨订单选择） → 填写发货数量 → 勾选 → 点击发货
    ↓
后端校验:
    批量查金蝶 SQL Server（ErpMapper.selectBatchRemainQty）获取每个物料的"剩余收料数量"
    校验: 本次发货数量 ≤ 剩余收料数量
    超发 → 返回 40000 参数错误
    ↓
步骤1: 所有物料分录合并为一次金蝶 Push 调用（支持跨订单），生成一张收料通知单
      写入本地 delivery_notice 表（status=1 发货中，存原始物料明细JSON）
      生成通知单号 SN+yyyyMMdd+序号（Redis INCR + 降级方案）
    ↓
步骤2: 下推成功 → 金蝶返回收料单内码(Id)和编号(Number) → 更新到本地 kingdee_receive_bill_id 和 kingdee_receive_bill_no
    ↓
步骤3: 调用金蝶 Save API（批量修改收料单数量为本次发货数量）
    ↓
全部成功 → 更新本地状态=已发货（status=2），设置 ship_time，返回所有送料单信息
部分失败 → 标记为已作废（status=3）
保存失败 → 调用金蝶 Cancel API 作废收料单，更新本地状态=已作废（status=3）或异常（status=4），记录 fail_reason
作废也失败(极端) → 记录异常日志，后台人工处理
```

### 7.2 送料通知单列表查询流程

```
查本地 delivery_notice 表（按 supplier_id 过滤）
    → 获取全部 kingdee_receive_bill_id 列表
    → 失败记录直接从 material_data JSON 读取
    ↓
查金蝶 SQL Server（按 kingdee_receive_bill_id 列表查询收料通知单物料明细）
    → 成功记录合并本地状态信息（发货时间、状态、失败原因）
    ↓
返回前端
    ↓
降级策略：金蝶数据库不可用时，成功记录返回空数组，附加 warning 提示
```

## 十一、技术要点

### 11.1 事务管理

发货流程涉及多次金蝶 API 调用和本地写操作，使用 `@Transactional` 管理本地事务：

```java
@Transactional(rollbackFor = Exception.class)
public DeliveryNoticeResp submitDelivery(DeliverySubmitReq req) {
    // 1. 校验数量
    // 2. 写本地 delivery_notice 表（status=1 发货中）
    // 3. Push API
    // 4. Save API
    // 5. 更新状态为 2（已发货）
    // 失败时 catch 中调用 Cancel API 作废 + 更新状态 + 记录原因
}
```

### 11.2 金蝶 SDK 线程安全

金蝶 SDK 的 `K3CloudApi` 非线程安全，每次调用应新建实例或使用线程池隔离。

### 11.3 送料通知单号生成规则

格式：`SN + yyyyMMdd + 4位序号`

示例：`SN202403200001`

序号按天重置，可考虑使用 Redis INCR 保证分布式环境下的唯一性。

### 11.4 JWT Token 设计

- 供应商 token：携带 `supplierId`、`username`、`role=supplier`
- 管理员 token：携带 `wxUserId`、`name`、`role=admin`
- Token 有效期：建议 24 小时

### 11.5 并发安全

- 发货数量校验：以金蝶 SQL Server 实时数据为准，避免本地缓存导致超发
- 同一分录并发发货：依赖金蝶侧剩余数量校验做兜底
- 送料通知单号生成：使用 Redis INCR 保证唯一性
