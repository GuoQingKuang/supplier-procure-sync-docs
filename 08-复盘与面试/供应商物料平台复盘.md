# 供应商物料平台 - 项目复盘

> 本文档是对 ProcureSync（供应商物料平台）项目的技术复盘，记录了项目现状、潜在优化方案及取舍思考过程。用于面试准备和个人技术沉淀。

## 项目概览

**ProcureSync** 是一个基于 Spring Boot 3.1.7 + Java 17 的采购数据同步系统，主要功能包括：

- 供应商采购订单查询（对接金蝶 ERP SQL Server）
- 送料通知单管理（发货流程）
- 预发货通知单管理（草稿→转正→提交）
- 企业微信 OAuth2 SSO 登录 + Keycloak OIDC 管理员认证
- MinIO 文件管理、阿里云短信服务、钉钉打卡同步

### 技术栈

| 组件 | 说明 |
|-----|------|
| Spring Boot 3.1.7 + Java 17 | 基础框架 |
| MyBatis-Plus 3.5.5 | ORM 框架 |
| 多数据源（MySQL + SQL Server） | 业务库 + 金蝶 ERP 只读查询 |
| Redis + Redisson | 缓存、分布式锁、单号生成 |
| 金蝶 K3Cloud SDK | 采购订单→收料通知单全链路对接 |
| JWT + OAuth2 (Keycloak) | 双角色认证（supplier/admin） |
| SpringDoc OpenAPI | API 文档 |

### 核心架构

```
Controller → Service → Dao(ServiceImpl) → Mapper(BaseMapper)
```

- **多数据源隔离**：MySQL 业务库与 SQL Server ERP 库分离，通过 `@DS("erp")` + 自注入绕过事务绑定
- **分布式锁**：Redis `setIfAbsent` 防并发转正/拆分/批量发货
- **单号生成**：Lua 脚本 `INCR + EXPIRE` 原子操作，失败降级到时间戳方案

## 核心业务流程

### 发货流程（两条路径）

**路径一：直接发货**
校验 ERP 剩余 → 写本地记录(status=发货中) → 金蝶 Push 下推 → 金蝶 Save 改交货数量 → 成功则 status=已发货

**路径二：预发货转正**
校验(ERP 剩余 - 草稿占用) → 生成本地草稿 → 编辑调整 → 分布式锁保护下转正 → 复用直接发货逻辑

**核心公式**：
- `ERP剩余 = FQTY - FBaseJoinQty`（不使用金蝶 FREMAINRECEIVEQTY，因为审批后才更新会偏高）
- `可用数量 = ERP剩余 - 同供应商所有 DRAFT 草稿的 deliveryQty 之和`

### 数量体系

| 场景 | 校验公式 |
|------|----------|
| 订单列表展示 | 纯 ERP 剩余，仅展示不校验 |
| 预发货 / 编辑 | `deliveryQty <= ERP剩余 - 草稿占用` |
| 正式发货 submit | `deliveryQty <= ERP剩余`（不扣草稿，金蝶 Push 后 FBaseJoinQty 自然更新兜底） |

## 深挖问题与优化方案

### 1. 批量发货同步阻塞

#### 当前状态

批量发货接口同步循环调用单个发货逻辑，平均响应 12 秒（单个 6-7 秒，一个订单可能拆成十几个子单）。前端用全屏加载覆盖等待，60 秒超时兜底。

压测数据：300 并发、平均响应 12 秒，全部请求成功。供应商最多 300 多家。

#### 考虑过的方案

**方案A：完全异步化（返回 taskId，前端轮询进度）**

- 做法：接口立即返回 taskId，后端线程池异步处理，前端轮询 `/progress/{taskId}` 查进度
- **舍弃原因**：业务要求用户必须等待完成才能操作下一个单。金蝶不允许多人同时操作同一物料行，同时操作会导致并发冲突。异步化让用户自由操作反而引发问题。

**方案B：Tomcat 线程转移（Future.get() 阻塞，释放 Tomcat 线程到自定义线程池）**

- 做法：提交到自定义线程池，Tomcat 线程 `Future.get(timeout)` 等待结果，超时则取消
- **舍弃原因**：增加代码复杂度但实际收益低。当前压测 300 并发全部成功，供应商仅 300 多家，Tomcat 线程池（默认 200 线程 + 100 队列）不会被打满。

#### 后续优化方案

当前配置够用，**不建议过度优化**。如果未来业务量增长导致 Tomcat 线程池频繁打满，优先考虑以下方案：

1. **增加限流保护**：限制同时执行的批量发货任务数量，避免突发流量打满线程池
2. **增加超时控制**：后端设置 50 秒超时（前端 60 秒），超时后主动中断并返回失败原因
3. **批量任务分批执行**：将大批量请求拆分成小批次，避免单次请求耗时过长

**面试话术**：

> "目前批量发货是同步调用，平均耗时 12 秒，压测 300 并发全部成功。考虑到供应商只有 300 多家，实际并发量远低于这个数字，所以当前方案是够用的。如果未来业务量增长到需要优化，会优先考虑加限流保护系统，而不是盲目异步化，因为业务本身要求用户等待，异步化的收益有限。"

### 2. 限流保护

#### 当前状态

当前项目没有限流保护。原因是供应商只有 300 多家，压测 300 并发、平均响应 12 秒的情况下全部请求都成功了，说明当前容量是够用的。

#### 考虑过的方案

**方案A：固定窗口计数器**

- 做法：每秒/每分钟最多 N 个请求，超过拒绝
- **舍弃原因**：存在窗口边界突刺问题。在窗口切换的瞬间，前一个窗口末尾 + 后一个窗口开头，实际瞬时 QPS 可能是限流值的 2 倍。

**方案B：滑动窗口（Redis ZSET + Lua）**

- 做法：把窗口分成多个小格子，每次统计最近 N 个格子的请求总数。用 ZSET 存请求时间戳，`ZREMRANGEBYSCORE` 清理过期数据，`ZCARD` 统计当前数量
- **舍弃原因**：精度够了，但引入 Redis 依赖较重，且当前量级根本不需要。

**方案C：令牌桶（Guava RateLimiter）**

- 做法：按固定速率往桶里放令牌，请求拿到令牌才放行。支持突发流量
- **舍弃原因**：当前量级不需要，但这是如果真要加限流的首选方案（改动最小、零外部依赖）。

**方案D：漏桶**

- 做法：固定大小队列 + 固定速率消费者。强制匀速处理，不允许突发
- **舍弃原因**：与令牌桶相比更适合"严格限流"场景，但当前项目不需要。

#### 后续优化方案

如果后续流量增长，会分阶段加：

**第一阶段**：先用 Guava RateLimiter 在核心接口（批量发货、批量提交）加本地令牌桶限流。改动最小，几行代码，能挡住明显的突发流量。

**第二阶段**：如果项目扩展到多机部署，单机限流不够用了，会考虑在 API 网关层（比如 Nginx）统一限流，或者引入 Sentinel 做分布式限流 + 熔断降级。

**第三阶段**：如果业务量再大到需要考虑精细化限流规则（比如按供应商维度、按接口维度动态配置），才会考虑 Redis + Lua 脚本的滑动窗口方案。

限流阈值的设定会基于压测数据：先压测出接口的极限 QPS，然后设到极限值的 70%~80%，留一定的缓冲余量。

### 3. 熔断降级（金蝶 API 调用）

#### 当前状态

当前金蝶 API 调用没有熔断机制。金蝶作为公司最重要的系统之一，稳定性很高，目前未出现过问题。

但一旦金蝶出现问题（挂了或响应极慢），会持续拖垮请求线程：
- 每个请求都要等到超时（比如 30 秒）才返回失败
- Tomcat 线程被占着不释放，大量请求堆积导致线程池满
- **整个系统不可用**（不只是金蝶相关接口，其他接口也受影响）

#### 考虑过的方案

引入 Resilience4j 或 Sentinel 做熔断降级。金蝶连续失败 N 次后熔断器打开，后续请求立即返回失败，过一段时间后自动半开试探恢复。

#### 后续优化方案

**短期应对（熔断 + 快速失败）**：
引入 Resilience4j 或 Sentinel 做熔断。金蝶连续失败 N 次后熔断器打开，后续请求立即返回"金蝶系统异常，请稍后重试"，而不是让请求一直等到超时。这样至少能保证系统其他功能（比如订单查询、预发货草稿编辑）不受影响。

**中期应对（异步补偿）**：
对于已经提交的发货请求，如果金蝶临时不可用，可以先将请求记录到本地队列（数据库或消息队列），等金蝶恢复后自动重试。用户侧显示"处理中"，避免直接报错。

**长期考虑（解耦 + 降级）**：
预发货单的草稿编辑、转正操作本身不依赖金蝶，可以正常进行。只有最终提交时才调金蝶。如果金蝶长时间不可用，可以考虑临时切换到"线下发货、事后补录"的模式，但这需要业务侧配合。

**面试话术**：

> "目前金蝶作为公司核心系统稳定性很高，熔断更多是防御性设计。但如果真要加，会优先考虑引入 Resilience4j 做熔断，连续失败后快速返回错误，避免拖垮整个系统。"

### 4. 超时控制

#### 当前状态

当前金蝶 API 调用、数据库查询没有设置明确的超时时间。前端设置了 60 秒超时兜底。

#### 后续优化方案

- **金蝶 API 调用**：设置 HTTP 超时时间 30 秒
- **数据库查询**：MyBatis 配置 `defaultStatementTimeout` 设置查询超时
- **批量发货整体**：后端设置 50 秒超时（前端 60 秒，后端提前 10 秒超时可避免前后端同时超时的竞态）

后端提前超时的意义：保证后端先返回明确的错误信息，前端有时间接收并展示，避免连接被中断导致前端收到不完整响应。

---

### 5. 可观测性（监控）

#### 当前状态

当前没有接口耗时监控、线程池运行状态监控、关键业务指标统计。出问题只能看日志排查，无法提前预警。

#### 后续优化方案

优先关注三个指标：

1. **金蝶 API 调用耗时**——最慢的外部依赖，平均 6-7 秒，一旦变慢能第一时间发现
2. **发货失败率**——按失败原因分类统计，金蝶侧问题集中爆发时能提前预警
3. **线程池队列长度**——排队数持续上升说明容量快不够了

实现上最简单的就是 AOP 拦截 Controller 记录耗时，日志里打上结构化 tag：

```java
log.info("metric_type=api|method={}|duration={}|status={}", methodName, duration, status);
```

后续可接 Prometheus + Grafana 做可视化，但当前阶段日志层面打出关键指标就够用。

**面试话术**：

> "目前出了问题只能看日志排查，效率低。如果加监控，会优先关注金蝶 API 耗时、发货失败率、线程池队列长度这三个指标。实现上最简单的就是 AOP 拦截 Controller 记录耗时，日志打 tag 方便后续统计。"

### 6. 线程池配置

#### 当前状态

- 业务线程池：核心 `CPU+1`，最大 `CPU×2+1`，队列 50，拒绝策略 `CallerRunsPolicy`
- 定时任务线程池：`CPU+1`，配置了优雅关闭，仅当 `app.enableScheduling=true` 时启用

#### 现状评估

- 线程池配置遵循 CPU 密集型公式，但短信发送是 IO 密集型，理论上线程数可以更大
- 定时任务线程池已配置但实际未使用（`ClockInSchedule.run()` 为空实现），属于过度设计
- 当前量级下配置完全够用，无需调优

**已记录**：线程池配置保持现状，当前量级足够，没有实际调优需求。

---

### 7. 数据库查询优化

#### 当前状态

金蝶 SQL 涉及多表 JOIN 查询，采用分页方式减少数据库压力：
- 先 `SELECT COUNT(*)` 计算总数
- 再 `SELECT ... LIMIT offset, pageSize` 返回当页数据

#### 现状评估

- 当前数据量下分页方式完全够用
- 如果数据量达到百万级，`COUNT(*)` 本身会变慢，可考虑近似总数或深度分页优化
- ERP 查询为只读操作，对主库无影响

---

### 8. 接口幂等性

#### 当前状态

- 批量发货、预发货转正等接口有分布式锁保护（防同时重复执行）
- 但锁释放后的重复请求仍会重复执行
- 业务状态校验有部分实现（如转正前检查 status=DRAFT）

#### 后续优化方案

**第一层：请求维度幂等**
前端提交时生成唯一 `requestId`，后端用 Redis 记录已处理的 requestId：
```java
SETNX processed:{requestId} {result} EX 86400
```
重复提交直接返回第一次的结果。

**第二层：业务状态校验**（已有部分）
发货前检查通知单状态，如果已经是"已发货"，说明之前已处理过，直接返回成功。

**面试话术**：

> "当前项目有分布式锁防止同时重复执行，但缺少请求维度的幂等控制。如果要完善，会加两层保护：前端传 requestId 做幂等键，后端用 Redis 记录已处理状态；同时做好业务状态校验，已发货的单不重复处理。"

## 面试高频考点速查

| 考点 | 当前方案 | 优化方向 | 面试话术要点 |
|-----|---------|---------|-------------|
| **批量发货同步阻塞** | 同步循环，12 秒响应 | 限流 + 超时控制 | 当前够用，不过度优化；优先限流而非异步化 |
| **限流保护** | 无 | Guava RateLimiter → Sentinel → Redis 滑动窗口 | 基于压测数据判断，分阶段加 |
| **熔断降级** | 无 | Resilience4j/Sentinel | 金蝶很少出问题，但熔断是保险丝 |
| **超时控制** | 前端 60 秒 | 后端 50 秒 + HTTP 30 秒 | 后端提前超时避免竞态 |
| **可观测性** | 日志排查 | AOP 耗时统计 + 结构化 tag | 优先监控最慢依赖和失败率 |
| **线程池** | CPU+1 公式 | 保持现状 | 量级够用，IO 密集型可更大但当前不需要 |
| **接口幂等性** | 分布式锁 + 部分状态校验 | requestId 幂等键 | 锁防并发，幂等防重复 |
| **JWT 失效** | Redis 黑名单 | 缩短 token 有效期 | JWT 无状态，黑名单是常见方案 |
| **多数据源** | @DS + 自注入 | 保持 | 自注入解决外层事务绑定问题 |
| **分布式锁** | setIfAbsent | Redisson RLock | 当前够用，生产建议用 Redisson |

## 总结与反思

### 做得好的地方

1. **分层架构清晰**：Controller → Service → Dao → Mapper，职责分明
2. **多数据源隔离**：MySQL 业务库与 SQL Server ERP 库分离，通过自注入解决事务绑定问题
3. **分布式锁实践**：Redis `setIfAbsent` 保证关键操作的幂等性
4. **失败回滚机制**：金蝶 API 失败后尝试删除收料单，状态标记为作废/异常
5. **数量校验严谨**：ERP 剩余 - 草稿占用的公式保证了发货数量不超过实际可发量

### 不足与改进

1. **缺少限流保护**：面对突发流量没有快速失败机制
2. **缺少熔断降级**：金蝶 API 不可用时会拖垮系统
3. **缺少超时控制**：慢调用可能无限期占用线程
4. **缺少可观测性**：出问题只能看日志，无法提前预警
5. **定时任务线程池未使用**：属于过度配置

### 如果重新做这个项目

1. **先压测再决定架构**：不凭感觉做优化，所有决策基于压测数据
2. **核心接口加限流**：用 Guava RateLimiter 快速接入
3. **金蝶 API 加熔断**：Resilience4j 几行代码搞定
4. **统一超时配置**：HTTP、数据库、业务层都设置明确超时
5. **AOP 记录耗时**：日志打 tag，方便后续接入监控系统

### 面试心态

面对"你的项目有什么优化空间"这类问题：

- **不要说"没有"**：任何项目都有优化空间，说没有显得思考不够
- **也不要说"我要全部重写"**：说明你对当前方案没信心
- **正确的姿态**：列出 2-3 个优化点，说明"当前够用，但如果量级增长我会这样加"
